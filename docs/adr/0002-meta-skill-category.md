# Introduce a `meta` category for skills that maintain this repo itself

`new-skill` (the skill that scaffolds other skills) doesn't fit either existing category: `engineering` is for skills that build and change a *target* codebase, `productivity` is for planning and non-code work, but `new-skill` acts on this skills repo's own structure — `plugin.json`, category docs, `skills-lock.json` — not on a project someone's building. We introduced `skills/meta/` for this and future repo-maintenance skills rather than force-fitting into `engineering` (mechanically similar — file creation, JSON edits — but conceptually different scope) or stretching `productivity`'s definition to cover repo tooling. This is hard to reverse once more meta skills accumulate under the category, and non-obvious — a future contributor could easily assume only two categories exist.
